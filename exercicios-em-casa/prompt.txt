PORQUE
Quando desenvolvemos um projeto (como um site ou aplicativo), usamos o Git para controlar e organizar as versões do código. Depois que o código está pronto ou atualizado, ele pode ser enviado ("implantado" ou "deployado") para rodar na nuvem, utilizando os serviços da AWS.
A Amazon Web Services (AWS) oferece mais de 200 serviços completos de datacenters em todo o mundo.

PORQUE
O uso do prompt de comando (terminal) com o Git permite realizar tarefas de versionamento de forma mais rápida e eficiente. Comandos digitados no terminal agilizam operações como salvar modificações, criar branches ou compartilhar código com o time.
A principal dificuldade inicial é aprender os comandos básicos do Git, mas, depois disso, o trabalho se torna muito mais produtivo e flexível -- ideal para equipes de desenvolvimento.

VERSIONADORES
Principais ferramentas de versionamento e plataformas que permitem o uso do Git para deploy de aplicações:
 GitHub  GitLab  Bitbucket  Azure DevOps (Repos)
Esses sistemas facilitam que desenvolvedores publiquem aplicações web, APIs, ou sites estáticos diretamente a partir dos repositórios Git, agilizando o processo de colocar o código atualizado em produção.

ENTENDENDO O GIT
· Git é um Sistema de gerenciamento de controle de versão descentralizado; · Possui recursos de monitoramento e acompanhamento de arquivos; · Uma vez ativo num projeto, o Git monitora as mudanças ocorridas no projeto e
nos mostra através de um Sistema linear em que fase se encontram os arquivos; · Status de arquivos: Untracked, tracked, modified e Staged.

ENTENDENDO O GIT
· Untracked: Quando um arquivo é adicionado num projeto monitorado; · Tracked: Quando o arquivo passa a ser monitorado através do commando git add; · Modified: Quando um arquivo monitorado é modificado; · Staged: Quando um arquivo monitorado é enviado ao repositório local (git commit). OBS.: Para passar um arquivo para Staged, este deve estar no estágio tracked, obrigatóriamente
PS.: Não execute nenhum commando desses ainda.

REALIZANDO COMMITS
Commit significa enviar seus arquivos e modificações para o repositório git local. Antes realizar o primeiro commit num projeto monitorado pelo git, você deve passar por 3 etapas:
· Configurar o nome de usuário;
· Configurar um email de usuário;
· Todos os arquivos a serem commitados precisam estar no estágio Tracked.
OBS.: Não é obrigatório, mas é considerada uma boa prática, inserir mensagens nos commits para identificá-los futuramente. Para inserir a mensagem no commit passando o parâmetro ­m "mensagem a ser enviada"

REALIZANDO COMMITS
Realizando as etapas do slide anterior:
· Configurar o nome de usuário: c "Seu Nome";
· Configurar o email do usuário: git config user.email "Seu Email";
· Adicionar os arquivos a serem commitados ao estágioTracked: git add "arquivo".
OBS.1: É muito recomendável que o e-mail que for configurado no projeto seja o mesmo e-mail que foi usado para criar a conta no repositório remote (no nosso caso, o GitHub)
OBS.2: A configuração de nome e email podem ser feita de formal global para que não precise ser informada a cada novo projeto. Isso tem seus pros e contras. Para configurar globalmente basta inserir o parâmetro --global depois de git config.

BRANCHES

TRABALHANDO COM BRANCHES
· Branches são ramificações do projeto;
· São usadas pra que vários desenvolvedores possam atuar em conjunto no mesmo projeto;
· Quando as alterações são feitas nas branches o ramo principal fica preservado até o conteúdo da nova branch ser `mergeado';
· Principais comandos: git branch (consulta branches do projeto) git branch nomeDaBranch (cria uma nova Branch) git checkout nomeDaBranch (muda para a nova Branch) git checkout ­b nomeDaBranch (cria uma nova branch e já muda pra ela)

GIT MERGE E GIT PULL
· Merge é uma operação feita para adicionar as modificações ao ramo principal do projeto;
· Git pull é um commando importantíssimo que todo desenvolvedor execcuta todo dia assim que começa a trabalhar. Ele verifica se há mudanças já implementadas no ramo principal e aplica na sua branch de trabalho para que você não trabalhe em cima de um Código desatualizado. O commando correto é sempre puxar do ramo principal (git pull origin master);

GIT STASH

GIT STASH
· Esse é um comando que ajuda muito quando nos atrapalhamos nessa salada de branches;
· Se você começar a desenvolver uma funcionalidade na branch errada, ele armazena temporariamente as modificações em memória e assim você consegue mudar de branch e puxar as modificações para a branch correta.
Para Aplicar devemos: - passar os arquivos para tracked (git add .); - digitar git stash para armazená-los em memória; - mudar para a branch correta; - digitar git stash pop para puxar as alterações da memória e aplicá-las na
